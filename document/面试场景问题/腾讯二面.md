# Java 面试题整理（资深开发视角）

### 1. 在类里定义一个和JDK原生类完全相同的类，比如String类，类名包名都一样，他能不能生效
**答：** 不能。基于JVM的双亲委派模型，加载`java.lang.String`时会先委托给最顶层的**Bootstrap ClassLoader（启动类加载器）**，该类加载器优先加载`$JAVA_HOME/lib`下的核心rt.jar。自定义的String类根本不会被加载，这是JVM的安全机制，防止核心API被篡改。除非自定义类加载器并强行打破双亲委派（严禁在生产环境这么做）。

### 2. 类加载器有哪些
**答：**
*   **Bootstrap ClassLoader（启动类加载器）：** C++实现，加载核心库（rt.jar）。
*   **Extension ClassLoader（扩展类加载器）：** 加载`jre/lib/ext`目录下的类。
*   **Application ClassLoader（应用程序类加载器）：** 加载ClassPath下的应用类，也叫系统类加载器。
*   **User ClassLoader（自定义类加载器）：** 继承`ClassLoader`，用于热部署、模块化等场景。

### 3. 线程池的构造参数有哪些
**答：** `ThreadPoolExecutor`的7个核心参数：
*   `corePoolSize`：核心线程数。
*   `maximumPoolSize`：最大线程数。
*   `keepAliveTime`：空闲线程存活时间。
*   `unit`：时间单位。
*   `workQueue`：阻塞队列（如`LinkedBlockingQueue`）。
*   `threadFactory`：线程工厂（用于命名线程）。
*   `handler`：拒绝策略。

### 4. 拒绝策略有哪些
**答：**
*   **AbortPolicy：** 默认策略，直接抛`RejectedExecutionException`异常。
*   **CallerRunsPolicy：** 由调用者线程（提交任务的线程）执行，减缓提交速度。
*   **DiscardPolicy：** 直接丢弃任务，不抛异常。
*   **DiscardOldestPolicy：** 丢弃队列头部（最老）的任务，然后重试提交。

### 5. 线程的生命周期
**答：** 共6种状态（`Thread.State`）：
1.  **NEW**：新建未启动。
2.  **RUNNABLE**：运行中或就绪（等待CPU调度）。
3.  **BLOCKED**：阻塞，等待进入`synchronized`临界区。
4.  **WAITING**：无限等待，如调用`Object.wait()`、`LockSupport.park()`。
5.  **TIMED_WAITING**：超时等待，如`sleep(long)`、`wait(long)`。
6.  **TERMINATED**：执行完毕。

### 6. JVM内存模型
**答：**
*   **线程私有：** 程序计数器、虚拟机栈、本地方法栈。
*   **线程共享：** 堆（存放对象实例）、方法区/元空间（存放类元数据、常量池）。
*   **直接内存：** NIO使用的堆外内存。

### 7. 堆内存怎么分的
**答：**
*   **新生代（Young）：** 占1/3堆空间。
    *   **Eden区：** 新对象分配区。
    *   **Survivor区（S0/S1）：** 两个大小相等的幸存区，用于复制回收。
*   **老年代（Old）：** 占2/3堆空间。存放长生命周期对象。

### 8. 新生代用什么垃圾回收机制
**答：** **复制算法（Copying）**。将Eden区和Survivor From区中存活的对象复制到Survivor To区，然后清空Eden和Survivor From。优点是不产生内存碎片，缺点是浪费了10%的Survivor空间。

### 9. CMS垃圾回收过程一共发生几次停顿，发生在哪里
**答：** **2次STW（Stop-The-World）停顿**。
过程：初始标记（STW，标记GC Roots） -> 并发标记 -> **重新标记（STW，修正并发期间的变动）** -> 并发清除。

### 10. HashMap的储存结构和怎么扩容
**答：**
*   **结构：** JDK 1.8后为**数组 + 链表 + 红黑树**。链表长度 >= 8且数组长度 >= 64时，链表转红黑树。
*   **扩容：** 默认负载因子0.75，容量翻倍（oldCap << 1）。JDK 1.8优化了重哈希逻辑，元素要么在原位置，要么在`原位置 + 旧容量`位置，无需重新计算hash。

### 11. 登录功能，如何保证密码储存安全和传输安全
**答：**
*   **传输安全：** 强制HTTPS（TLS加密），防止中间人攻击。
*   **存储安全：** **禁止明文存储**。使用**BCrypt、PBKDF2**等带随机盐（Salt）的慢哈希算法加密存储，即使数据库泄露也难以反向破解。
*   **防御：** 登录失败锁定、验证码、密码复杂度校验。

### 12. redis想实现热门商品排行榜，用哪种数据结构
**答：** **Sorted Set (ZSet)**。将商品ID作为`member`，热度值（销量/点击量）作为`score`。使用`ZINCRBY`增加热度，`ZREVRANGE`获取Top N榜单。

### 13. zset底层的储存结构
**答：**
*   **ziplist（压缩列表）：** 元素少（<128）且体积小（<64字节）时使用，节省内存。
*   **skiplist（跳表）：** 不满足ziplist条件时使用，支持平均O(logN)查询和范围查询。

### 14. redis有哪几种持久化策略
**答：**
*   **RDB：** 定时快照。恢复快，文件小，但可能丢数据。
*   **AOF：** 记录写命令日志。数据安全高（秒级同步），文件大，恢复慢。
*   **混合持久化（4.0+）：** RDB文件头 + AOF增量数据。兼顾RDB的快速恢复和AOF的数据安全。

### 15. 项目中，后端提供的接口，可能抛异常，要对前端做一个提示“系统繁忙稍后再试”，怎么做
**答：** 使用**`@ControllerAdvice` + `@ExceptionHandler`**做全局异常处理。定义统一返回体（如`Result`类），在异常捕获方法中记录详细错误日志（Log.error），并向前端返回`{"code": 500, "msg": "系统繁忙稍后再试"}`。避免将具体异常堆栈暴露给前端。
